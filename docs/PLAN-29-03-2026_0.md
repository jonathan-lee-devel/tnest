# tnest Implementation Plan

**Date:** 29 March 2026
**Package:** `@jdevel/tnest`
**Goal:** Type-safe communication between NestJS microservices via shared contracts, enforced at compile time.

---

## Current State

The codebase is a scaffold with a single placeholder module and service. No contracts, decorators, typed client, or transport integration exist yet.

---

## Phase 1 — Contract System (Core Types)

The foundation of the library: the type primitives that define message contracts.

### 1.1 Define contract interfaces

- `Command<TPattern, TPayload, TResponse>` — a request/response message (one-to-one).
- `Event<TPattern, TPayload>` — a fire-and-forget message (one-to-many).
- `Query<TPattern, TPayload, TResponse>` — semantically distinct from Command; read-only request/response.
- All three carry the message pattern (topic/route), payload shape, and (where applicable) response shape as generic parameters.

### 1.2 Contract registry type

- `ContractRegistry` — a mapped type that acts as a single source of truth. Consumers define an interface like:
  ```ts
  interface MyContracts {
    'user.create': Command<'user.create', CreateUserDto, User>;
    'user.created': Event<'user.created', UserCreatedPayload>;
    'user.get': Query<'user.get', { id: string }, User>;
  }
  ```
- Utility types to extract patterns, payloads, and responses from a registry:
  - `PatternOf<C>`, `PayloadOf<C>`, `ResponseOf<C>`
  - `CommandsOf<Registry>`, `EventsOf<Registry>`, `QueriesOf<Registry>`

## Phase 2 — Typed Client (Producer Side)

A wrapper around NestJS `ClientProxy` that enforces contract types when sending messages.

### 2.1 `TypedClient<TRegistry>`

- Generic service that wraps `ClientProxy`.
- Methods:
  - `send<P extends CommandPatterns<TRegistry>>(pattern: P, payload: PayloadOf<TRegistry[P]>): Observable<ResponseOf<TRegistry[P]>>` — for commands/queries.
  - `emit<P extends EventPatterns<TRegistry>>(pattern: P, payload: PayloadOf<TRegistry[P]>): Observable<void>` — for events.
- Pattern argument is constrained to the union of registered patterns, payload and response are inferred.

### 2.2 `TypedClientFactory`

- Injectable factory that creates `TypedClient` instances from a `ClientProxy` (or `ClientProxyFactory` options).
- Allows multiple named clients (e.g. one per remote service).

### 2.3 Files

- `src/client/typed-client.ts`
- `src/client/typed-client-factory.ts`
- `src/client/index.ts`

## Phase 3 — Typed Handlers (Consumer Side)

Decorators that enforce contract types on message handlers.

### 3.1 `@TypedMessagePattern<TRegistry, P>(pattern)`

- Wraps `@MessagePattern` from `@nestjs/microservices`.
- Constrains the handler's parameter type to `PayloadOf<TRegistry[P]>` and return type to `ResponseOf<TRegistry[P]>`.
- Uses TypeScript decorator metadata so the compiler catches mismatches.

### 3.2 `@TypedEventPattern<TRegistry, P>(pattern)`

- Wraps `@EventPattern`.
- Constrains handler parameter to `PayloadOf<TRegistry[P]>`, return type `void`.

### 3.3 Handler type helpers

- `TypedMessageHandler<TRegistry, P>` — function type signature for a command/query handler.
- `TypedEventHandler<TRegistry, P>` — function type signature for an event handler.
- These can be used even without the decorators, for consumers who prefer manual typing.

### 3.4 Files

- `src/handlers/typed-message-pattern.decorator.ts`
- `src/handlers/typed-event-pattern.decorator.ts`
- `src/handlers/handler-types.ts`
- `src/handlers/index.ts`

## Phase 4 — Module Integration

Wire everything into a proper NestJS dynamic module.

### 4.1 `TnestModule.forRoot(options)`

- Accepts configuration:
  - `clients` — array of `{ name: string; transport: Transport; options: ClientOptions }` to register named typed clients.
- Registers `TypedClientFactory` as a provider.
- Registers each named client as an injectable token.

### 4.2 `TnestModule.forRootAsync(options)`

- Standard NestJS async module factory (`useFactory`, `useClass`, `useExisting`, `inject`).

### 4.3 Injection tokens

- `TNEST_OPTIONS` — module configuration.
- Per-client tokens via `getClientToken(name)` helper.

### 4.4 Remove placeholder service

- Delete current `TnestService.greet()`.
- Replace `TnestService` with the real typed client logic, or remove it entirely if `TypedClient` + factory replaces its role.

### 4.5 Files

- Update `src/tnest.module.ts`
- `src/constants.ts` (injection tokens)
- `src/interfaces/module-options.ts`
- Update `src/index.ts` (barrel exports)

## Phase 5 — Serialization & Validation (Optional Runtime Layer)

An opt-in runtime validation layer for consumers who want runtime safety in addition to compile-time safety.

### 5.1 Contract validation decorator

- `@ValidateContract()` — method decorator that validates incoming payloads against a schema at runtime.
- Pluggable validator interface so users can bring their own (class-validator, zod, joi, etc.).

### 5.2 Serialization hooks

- `PayloadSerializer` / `PayloadDeserializer` interfaces.
- Default passthrough implementation; users can provide custom serialization (e.g. protobuf, msgpack).
- Registered via module options.
