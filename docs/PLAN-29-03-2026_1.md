# tnest Improvement Plan — Quality, Testing & CI

**Date:** 29 March 2026
**Package:** `@jdevel/tnest@0.0.2`
**Goal:** Harden the library with comprehensive testing, fix lint issues, set up CI, and address gaps in the current implementation.

---

## Current State

All core functionality from Plan 0 is implemented and builds cleanly:

- Contract types (Command, Event, Query) with utility types and builder helpers
- TypedClient / TypedClientFactory (producer side)
- TypedMessagePattern / TypedEventPattern decorators (consumer side)
- TnestModule with `forRoot` / `forRootAsync`
- Optional runtime validation and serialization
- MockTypedClient and TestContractModule (testing utilities)
- ESLint configured, README and USAGE docs written

**What's missing:**

- No test runner or tests
- No CI pipeline
- No formatter (Prettier)
- 45 ESLint warnings (0 errors)
- `tnest.service.ts` was deleted but CLAUDE.md still references it as the architecture
- `forRootAsync` has a gap — individual client providers are not dynamically registered/exported when using async options (they're computed inside a factory but never materialized as injectable tokens)
- The lint script in `package.json` uses bare `eslint` instead of `./node_modules/.bin/eslint` (or `npx`)

## Phase 1 — Test Runner Setup

- `jest` + `ts-jest` — test runner with TypeScript support
- `@types/jest` — type definitions
- `@nestjs/testing` — NestJS test utilities (needed for module/integration tests)

  - Test match: `**/*.spec.ts`

### 1.3 Add npm scripts

- `test` — run Jest
- `test:watch` — run Jest in watch mode
- `test:cov` — run Jest with coverage

- `jest.config.ts`
- Update `package.json` (devDependencies + scripts)

## Phase 2 — Compile-Time Type Tests

Verify that the type system — the core value proposition of the library — works correctly. These tests use `expect-type` to assert type relationships at compile time.

### 2.1 Contract type tests

- `src/contracts/__tests__/contract-types.spec.ts`
- Assert `Command`, `Event`, `Query` produce the correct branded types.
- Assert `PatternOf`, `PayloadOf`, `ResponseOf` extract the right types.
- Assert `CommandsOf`, `EventsOf`, `QueriesOf` correctly filter a registry.
- Assert `CommandPatterns`, `EventPatterns`, `QueryPatterns` produce correct string literal unions.
- Assert `ValidateRegistry` accepts valid registries and rejects mismatched ones.

### 2.2 Define-helpers type tests

- `src/contracts/__tests__/define-helpers.spec.ts`
- Assert `defineRegistry` infers correct pattern keys and contract types.
- Assert `command()`, `event()`, `query()` produce correctly typed contracts.

### 2.3 Client type tests

- `src/client/__tests__/typed-client-types.spec.ts`
- Assert `TypedClient.send()` accepts only command/query patterns and infers payload/response.
- Assert `TypedClient.emit()` accepts only event patterns and infers payload.
- Assert passing the wrong pattern or payload type is a compile error.

### 2.4 Handler type tests

- `src/handlers/__tests__/handler-types.spec.ts`
- Assert `TypedMessageHandler<R, P>` produces the correct function signature.
- Assert `TypedEventHandler<R, P>` produces the correct function signature.

## Phase 5 — Fix `forRootAsync` Client Registration Gap

The current `forRootAsync` implementation computes client providers inside a factory but doesn't actually register them as injectable providers. The `TNEST_CLIENT_PROVIDERS` factory runs but its return value (an array of providers) is never added to the module's provider list.

### 5.1 Fix the async client registration

Two options:

**Option A (Recommended):** Use `onModuleInit` lifecycle hook. After `TNEST_OPTIONS` resolves, dynamically create and register client `ClientProxy` instances using `ModuleRef`.

**Option B:** Require users to declare client names upfront in async options so tokens can be statically registered, with factories that depend on `TNEST_OPTIONS`.

### 5.2 Add integration test proving async clients are injectable

- Test that a `TnestModule.forRootAsync()` with client definitions results in injectable client tokens.

### 5.3 Files

- Update `src/tnest.module.ts`
- Update `src/interfaces/module-options.ts` (if Option B)
- Test in `src/__tests__/tnest-module.spec.ts`

```
                                  |                       |
                            Phase 6 (Lint/Quality)  Phase 7 (CI)
                                  |                       |
                                  +-----------+-----------+
                                              |

```

- Phase 1 is the prerequisite for all test phases.
- Phases 2 and 3 can be done in parallel once Phase 1 is complete.
- Phase 4 depends on the test runner but is otherwise independent.

- Phases 6 and 7 can be parallelized.
- Phase 8 is a final cleanup pass.
